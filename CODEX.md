# TeamFlow Coding Conventions

이 문서는 TeamFlow MVP 구현 시 지켜야 할 기본 코딩 관례를 정리한다. 목표는 복잡한 규칙을 늘리는 것이 아니라, 현재 단계에서 일관된 구현 기준을 유지하는 것이다.

## 핵심 원칙
- 단순함을 우선한다. 과도한 추상화, 범용화, 조기 최적화를 피한다.
- 각 Phase는 항상 실행 가능한 상태를 유지한다.
- MVP 범위를 벗어나는 구조는 미리 도입하지 않는다.

## 공통 구조 규칙
- 도메인 패키지 내부는 역할별로 분리한다.
  - 예: `controller`, `service`, `repository`, `dto`, `entity`
- 여러 도메인에서 재사용되는 코드는 `common`으로 올리되, 성급하게 공통화하지 않는다.
- 엔티티, DTO, 서비스 로직, API 응답 모델을 한 파일에 섞지 않는다.

## 백엔드 규칙

### API/Controller
- REST + JSON 기준으로 구현한다.
- 컨트롤러는 요청 검증, 서비스 호출, 응답 반환에 집중한다.
- 비즈니스 로직은 컨트롤러에 넣지 않는다.
- 엔티티를 직접 응답으로 반환하지 않는다. 반드시 Response DTO를 사용한다.
- 요청/응답 DTO는 분리한다.
- 경로는 `spec.md`에 정의된 MVP API를 우선 기준으로 맞춘다.

### Service
- 비즈니스 규칙과 권한 검사는 서비스 계층에 둔다.
- 서비스 메서드 이름은 동작이 드러나게 작성한다.
  - 예: `login`, `createTask`, `updateTaskStatus`, `addShare`

### Repository
- Spring Data JPA 기본 기능으로 충분하면 커스텀 구현을 늘리지 않는다.
- 조회 메서드는 실제 화면/API 요구사항에 맞게 명확하게 만든다.

### Entity
- 엔티티는 도메인 상태와 관계를 표현하는 데 집중한다.
- 무한 참조나 직렬화 문제를 피하기 위해 API 응답에는 엔티티를 노출하지 않는다.
- 시간 필드(`createdAt`, `updatedAt`, `completedAt`)는 일관되게 관리한다.
- Enum은 문자열 의미가 명확하게 드러나도록 유지한다.

### DTO/Validation
- 요청 DTO에는 `jakarta.validation`을 적용한다.
- 필수값, 길이, 이메일 형식 등은 DTO에서 먼저 검증한다.
- 검증 실패 응답 형식은 전역 예외 처리 기준에 맞춘다.
- DTO 이름은 역할이 드러나게 작성한다.
  - 예: `LoginRequest`, `TaskCreateRequest`, `TaskResponse`

### Exception
- 전역 예외 처리는 `@RestControllerAdvice`로 일원화한다.
- 예외 메시지는 사용자 관점에서 이해 가능하게 작성한다.
- 권한 없음은 403, 리소스 없음은 404, 검증 실패는 400으로 구분한다.

## 권한 규칙
- 작성자(creator): 수정, 삭제, 공유 가능
- 담당자(assignee): 상태 변경, 댓글 가능
- 공유 대상(shared user): 조회, 댓글 가능
- 권한 로직은 컨트롤러마다 중복하지 말고 서비스 또는 공통 권한 유틸로 일관되게 처리한다.
- 권한 없는 사용자의 상세 조회/수정/삭제는 반드시 차단한다.

## 데이터/샘플 규칙
- 개발과 데모 환경은 H2 in-memory 기준으로 유지한다.
- 샘플 데이터는 실행 직후 주요 흐름을 검증할 수 있는 수준으로 유지한다.
- 샘플 데이터 주입 방식은 `data.sql` 또는 `CommandLineRunner` 중 하나로 통일한다.
- 샘플 사용자와 태스크는 로그인, 조회, 공유, 댓글, 알림 시나리오를 재현할 수 있어야 한다.

## 프론트엔드 규칙

### 구조
- 프론트엔드는 `frontend/` 아래 Next.js App Router 기준으로 구성한다.
- 페이지는 `app/`, 재사용 컴포넌트는 `components/`, API 호출은 `lib/api/`, 타입은 `types/`로 분리한다.
- 페이지 파일에는 페이지 조합 로직을 두고, 반복 UI는 컴포넌트로 분리한다.

### UI/상태
- 핵심 UX는 빠른 입력과 리스트 중심 조회다. 불필요한 클릭 수를 늘리지 않는다.
- 로딩, 에러, 빈 상태를 명시한다.
- MVP 단계에서는 단순하고 유지보수 쉬운 상태 관리 방식을 우선한다.

### 스타일
- Tailwind CSS와 CSS 변수를 사용한다.
- 임의의 인라인 스타일 남발을 피한다.
- 색상, 간격, 반경, 텍스트 톤은 반복 사용 가능한 방식으로 맞춘다.
- 기존 화면 톤이 정해지면 이후 페이지도 같은 시각 언어를 따른다.

## 테스트/검증 규칙
- 핵심 서비스 로직에는 단위 테스트를 우선한다.
- 최소한의 API 검증 테스트를 유지한다.
- 기능 구현 후에는 해당 Phase 완료 기준을 기준으로 직접 검증한다.
- 버그를 수정했으면 가능하면 같은 유형을 막는 검증 코드를 함께 추가한다.

## 문서 규칙
- 실행 방법은 README를 기준 문서로 유지한다.
- API, 구현 범위, 작업 분해는 `spec.md`, `plan.md`, `tasks.md`와 일관되게 맞춘다.
- 구조나 정책이 바뀌면 코드보다 늦지 않게 문서도 함께 갱신한다.

## 커밋 규칙
- 한 커밋에는 한 가지 의도를 담는다.
- 커밋 메시지는 짧고 명확하게 작성한다.
- 문서, 구조 정리, 기능 구현, 리팩터링은 가능한 한 분리한다.
- 예시:
  - `feat: add login API`
  - `feat: implement quick task creation`
  - `refactor: split task dto packages`
  - `docs: update phase checklist`

## 금지 사항
- 엔티티 직접 응답
- 컨트롤러에 비즈니스 로직 누적
- 미사용 추상화 추가
- MVP 범위를 벗어난 인증/실시간/복잡한 권한 시스템 선도입
- 문서와 실제 구현 상태의 불일치 방치
